# 프롬프트 설계(Prompt Engineering)란?

프롬프트 설계란 생성형 AI(LLM, Large Language Model)에게 **원하는 결과를 얻기 위해 입력 문장(프롬프트)을 구조적으로 설계하는 기술**입니다.

즉, AI에게 단순히 질문만 하는 것이 아니라:

* 역할(Role)
* 목적(Objective)
* 출력 형식(Output Format)
* 제약 조건(Constraints)
* 예시(Examples)
* 단계별 사고 방식(Reasoning)

등을 함께 제공하여 AI가 더 정확하고 안정적으로 동작하도록 만드는 것입니다.

예를 들어:

단순 질문:

> Kubernetes 설명해줘

프롬프트 설계 적용:

> 너는 클라우드 인프라 강사이다.
> Kubernetes를 비전공자 기준으로 설명하고,
> Docker와 비교하며,
> 실무 예제 3개와 YAML 예시를 포함해줘.
> 마지막에는 핵심 요약 표를 만들어줘.

→ 결과 품질 차이가 매우 커집니다.

---

# 왜 프롬프트 설계가 중요한가?

LLM은 사람처럼 “이해”하는 것이 아니라:

* 다음 단어를 예측
* 문맥(Context) 기반 확률 계산
* Attention 기반 연관성 분석

으로 동작합니다.

따라서 입력 구조가 불명확하면:

* 엉뚱한 답변
* 형식 오류
* 환각(Hallucination)
* 불완전한 설명

이 발생할 가능성이 커집니다.

즉:

> “좋은 AI 결과 = 좋은 프롬프트 구조”

라고 볼 수 있습니다.

---

# 프롬프트 설계의 핵심 원리

대표적인 핵심 원리는 다음과 같습니다.

1. 명확성(Clarity)
2. 구체성(Specificity)
3. 문맥 제공(Context)
4. 역할 부여(Role Assignment)
5. 출력 형식 지정(Output Control)
6. 단계적 사고 유도(Chain of Thought)
7. 예시 기반 학습(Few-shot Prompting)
8. 제약 조건 설정(Constraints)
9. 반복 개선(Iterative Refinement)
10. 토큰 효율성(Token Efficiency)

---

# 1. 명확성(Clarity)

AI는 애매한 표현을 매우 어려워합니다.

## 나쁜 예시

> 서버 설명해줘

→ 어떤 서버인지 모름

* 웹 서버?
* DB 서버?
* Kubernetes API Server?
* 물리 서버?

## 좋은 예시

> Ubuntu 기반 Nginx 웹 서버의 역할과 동작 과정을 설명해줘

---

# 2. 구체성(Specificity)

조건을 구체적으로 줄수록 결과 품질이 향상됩니다.

## 예시

### 일반 질문

> Docker 설명해줘

### 구체적 질문

> Docker를 비전공 대학생 기준으로 설명하고,
> VM과 차이점 비교표를 포함하고,
> 실제 컨테이너 실행 명령어 예시도 작성해줘

---

# 3. 문맥(Context) 제공

LLM은 현재 문맥(Context)에 매우 의존합니다.

## 예시

### 문맥 없음

> 왜 느려?

→ 무엇이 느린지 모름

### 문맥 제공

> Kubernetes에서 FastAPI 기반 AI 추론 서버가
> GPU 사용률은 낮은데 응답 속도가 느린 이유를 설명해줘

→ 훨씬 정확해짐

---

# 4. 역할(Role Assignment)

AI에게 역할을 부여하면 결과 스타일이 달라집니다.

## 예시

### 일반

> Kubernetes 설명해줘

### 역할 지정

> 너는 DevOps 전문가이다.
> Kubernetes를 운영 관점에서 설명해줘

또는:

> 너는 대학 교수이다.
> 비전공자에게 쉽게 설명해줘

---

# 5. 출력 형식(Output Control)

원하는 출력 형식을 지정하는 것이 매우 중요합니다.

## 예시

> 다음 형식으로 출력해줘:
>
> 1. 개념
> 2. 구조
> 3. 장단점
> 4. 실무 예제
> 5. YAML 예시
> 6. 핵심 요약표

---

# 6. 단계적 사고 유도 (Chain of Thought)

AI가 문제를 단계별로 생각하게 만드는 기법입니다.

복잡한 문제일수록 매우 중요합니다.

## 예시

### 일반 질문

> Kubernetes 네트워크 문제 해결해줘

### 단계적 사고 유도

> 아래 순서로 분석해줘:
>
> 1. Pod 상태 확인
> 2. Service 확인
> 3. DNS 확인
> 4. CNI 확인
> 5. iptables 확인
> 6. 해결 방법 제시

---

# 7. Few-shot Prompting (예시 기반)

예시를 먼저 보여주고 같은 패턴을 따르게 하는 방식입니다.

## 예시

### 입력

```text
입력: Docker
출력:
- 정의:
- 장점:
- 단점:

입력: Kubernetes
출력:
```

AI는 위 패턴을 따라 출력합니다.

---

# 8. 제약 조건(Constraints)

불필요한 출력 방지에 중요합니다.

## 예시

> 500자 이하로 설명해줘
> 코드만 출력해줘
> 표 형태로만 작성해줘
> Python 예제만 사용해줘
> 초보자 수준으로 설명해줘

---

# 9. 반복 개선(Iterative Refinement)

좋은 프롬프트는 한 번에 완성되지 않습니다.

보통:

1. 초안 생성
2. 결과 확인
3. 부족한 부분 수정
4. 프롬프트 개선

과정을 반복합니다.

---

# 10. 토큰 효율성(Token Efficiency)

LLM은 토큰 단위로 처리합니다.

너무 긴 프롬프트는:

* 비용 증가
* 속도 저하
* 문맥 손실

문제를 발생시킬 수 있습니다.

따라서:

* 핵심만 작성
* 중복 제거
* 구조화
* 불필요한 설명 제거

가 중요합니다.

---

# 프롬프트 구조의 대표 패턴

대표적인 구조:

```text
[역할]
너는 DevOps 전문가이다.

[목표]
Kubernetes Ingress를 설명해줘.

[대상]
비전공자 기준

[조건]
- YAML 포함
- 그림처럼 단계별 설명
- Nginx Ingress 기준

[출력형식]
1. 개념
2. 구조
3. 트래픽 흐름
4. YAML 예시
5. 운영 시 주의사항
```

# 프롬프트 설계 기법 종류

| 기법                  | 설명                      |
| --------------------- | ------------------------- |
| Zero-shot             | 예시 없이 바로 요청       |
| One-shot              | 예시 1개 제공             |
| Few-shot              | 예시 여러 개 제공         |
| Chain-of-Thought      | 단계적 사고 유도          |
| Self-Consistency      | 여러 추론 후 최적 답 선택 |
| ReAct                 | 추론 + 행동 결합          |
| Tree of Thoughts      | 여러 사고 경로 탐색       |
| Role Prompting        | 역할 기반 응답            |
| Instruction Prompting | 명령 기반                 |
| Contextual Prompting  | 문맥 기반                 |

---

# Zero-shot vs Few-shot

## Zero-shot

```text
Python으로 정렬 알고리즘 설명해줘
```

## Few-shot

```text
입력: Stack
출력:
- 정의
- 특징
- 예제

입력: Queue
출력:
```

Few-shot이 형식을 훨씬 잘 맞춥니다.

---

# Chain of Thought(CoT)란?

복잡한 문제를 단계적으로 생각하도록 만드는 기법입니다.

## 예시

### 일반

> 128GB RAM 서버에 Kubernetes Pod를 몇 개 배치할 수 있나?

### CoT

> 아래 순서로 계산해줘:
>
> 1. OS 예약 메모리
> 2. Kubernetes 시스템 메모리
> 3. Pod당 메모리
> 4. 최대 Pod 수 계산

---

# ReAct Prompting

ReAct = Reason + Act

AI가:

1. 생각(Reason)
2. 행동(Act)
3. 결과 확인(Observation)

을 반복합니다.

AI Agent 시스템에서 많이 사용됩니다.

---

# 실제 실무에서 중요한 프롬프트 원칙

실무에서는 특히 다음이 중요합니다.

| 중요도    | 원칙           |
| --------- | -------------- |
| 매우 높음 | 역할 지정      |
| 매우 높음 | 출력 형식 지정 |
| 매우 높음 | 예시 제공      |
| 매우 높음 | 단계별 추론    |
| 높음      | 제약 조건      |
| 높음      | 문맥 제공      |
| 중간      | 토큰 최적화    |

---

# AI가 잘못 답하는 이유

대표 원인:

| 원인             | 설명               |
| ---------------- | -----------------  |
| 애매한 질문      | 의미 불명          |
| 문맥 부족        | 상황 이해 실패     |
| 출력 형식 미지정 | 결과 구조 불안정   |
| 과도한 길이      | 문맥 손실          |
| 충돌하는 지시    | 서로 다른 요구사항 |
| 잘못된 예시      | 잘못 학습          |

---

# 좋은 프롬프트 특징

좋은 프롬프트는:

* 명확함
* 구조적임
* 목적이 분명함
* 출력 형식이 있음
* 제약 조건이 있음
* 대상 독자가 정의됨
* 예시가 있음
* 단계적임

---

# 실제 예시 (고급)

## AI 인프라 설계 요청

```text
너는 MLOps 아키텍트이다.

다음 조건으로 Kubernetes 기반 AI 추론 플랫폼을 설계해줘.

조건:
- GPU 서버 사용
- FastAPI 모델 서빙
- Prometheus/Grafana 모니터링
- GitLab CI/CD 자동배포
- ArgoCD GitOps
- 고가용성 구성

출력:
1. 전체 아키텍처
2. 네트워크 흐름
3. Helm 구조
4. YAML 예시
5. 운영 전략
6. 장애 대응 전략
```

# 프롬프트 설계의 본질

프롬프트 설계의 본질은:

> “AI가 정확하게 이해할 수 있도록 문제를 구조화하는 기술”

입니다.

즉:

* 질문 기술
* 문제 정의 기술
* 정보 구조화 기술
* 요구사항 명세 기술

이 모두 포함됩니다.

프롬프트 설계란:

> 생성형 AI가 원하는 결과를 안정적이고 정확하게 생성하도록 입력을 구조화하는 기술입니다.
