# OpenAI API Key 생성 시 “이름(Name)”의 의미

API Key 생성할 때 입력하는 Name은 이 API Key를 어디에 사용하는지 식별하기 위한 별칭(Label)입니다.

즉 **관리용 이름** 입니다.

---

# 1. 중요한 점

# 인증에는 사용되지 않음

예를 들어:

```text id="4vc4k0"
my-fastapi
```

라고 이름을 만들어도,

실제 인증은:

```text id="5c9oq8"
sk-xxxxxxxx
```

이 비밀키 자체로 합니다.

즉:

| 항목            | 역할                  |
| --------------- | --------------------- |
| Name            | 사람이 보기 위한 이름 |
| API Key(sk-...) | 실제 인증 키          |

입니다.

---

# 2. 왜 이름이 필요한가?

프로젝트가 많아지면 API Key도 많아집니다.

예:

| 이름            | 용도        |
| --------------- | ----------  |
| fastapi-chatbot | 챗봇 서버   |
| gradio-test     | 테스트 UI   |
| kubernetes-prod | 운영 서버   |
| local-llm-proxy | 내부 프록시 |
| mobile-app      | 모바일 앱   |

---

# 3. 실제 사용 상황

예를 들어 사용자가:

* FastAPI 서버
* Kubernetes 운영
* Gradio 테스트
* 로컬 개발
* 운영 환경

을 각각 사용하면,

키를 분리하는 것이 좋습니다.

---

# 4. 왜 분리하는가?

## (1) 보안

특정 키 유출 시 해당 키만 삭제 하면 됩니다
---

## (2) 사용량 추적

예:

```text id="03u9kr"
어떤 프로젝트가 비용을 많이 쓰는지 확인
```

가능.

---

## (3) 환경 분리

| 환경          | 키        |
| ------------  | --------- |
| 개발(dev)     | dev-key   |
| 테스트(stage) | stage-key |
| 운영(prod)    | prod-key  |

---

# 5. 추천 이름 규칙

실무에서는 보통:

```text id="evrjrf"
프로젝트명-환경명
```

형태 사용.

---

## 예시

```text id="u4u8qt"
fastapi-dev
fastapi-prod
gradio-local
k8s-chatbot
rag-system
```

---

# 6. 절대 포함하면 안 되는 것

이름에:

* 비밀번호
* 개인정보
* 고객명
* 카드번호

같은 민감정보 넣지 말 것.

---

# 7. 실제 인증 흐름

예를 들어 Name이 `fastapi-prod` 라고 만들어도 실제 요청에서는 Authorization: Bearer sk-xxxx 만 사용됩니다.

즉 서버는 Name은 거의 신경 안 씀

---

# 8. 쉽게 비유하면

| 개념    | 비유             |
| ------- | ---------------- |
| Name    | 열쇠에 붙인 메모 |
| API Key | 실제 열쇠        |

즉:

```text id="8llm1d"
"사무실 키"
"집 키"
```

처럼 구분하는 용도입니다.
